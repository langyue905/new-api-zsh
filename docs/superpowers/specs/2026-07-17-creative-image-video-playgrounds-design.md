# 创作中心图像与视频 Playground 集成设计

## 1. 目标与范围

第一阶段把两个开源 Playground 合入现有 New API classic 前端，使登录用户可以在同一套页面中访问图像生成和视频生成：

- 图像：`CookSleep/gpt_image_playground`，MIT 协议。
- 视频：`alasano/sora-2-playground`，MIT 协议。
- 侧边栏“创作”下显示“游乐场 / 图像 / 视频”。
- 新增受登录保护的 `/console/image` 和 `/console/video` 页面。
- 两个页面都默认请求当前 New API 的同源 `/v1` 接口，不增加常驻服务或新容器。
- 第一阶段暂时保留 Playground 的手动 API 地址/密钥配置；用户确认后的“首次点击自动创建并复用专用密钥”属于第二阶段，不在本次实现范围内。

不在本阶段实现：自动密钥创建、密钥权限分组、后台密钥回收策略、独立域名、计费规则改造，以及与现有聊天 Playground 无关的重构。

## 2. 方案与架构

采用原生集成，而不是 iframe 或独立站跳转。两个项目的页面组件、状态管理和必要的 API 适配层进入 `web/classic`，由现有 RSBuild 构建并随 New API 一起发布。

页面壳由 New API 的 `PageLayout` 提供，Playground 只负责内容区，避免重复顶部栏、登录页、侧边栏和主题。路由组件使用懒加载，避免用户打开普通控制台页面时加载图像/视频编辑器的大型依赖。

原项目中的 Next.js/Vite 应用壳、独立启动脚本和服务端 API 路由不直接部署。视频 Playground 的服务层改为调用 New API 的 OpenAI 兼容视频接口；图像 Playground 的 OpenAI 兼容客户端沿用同样的同源配置。保留原项目的 MIT 版权和必要的来源说明。

## 3. 页面与组件边界

### 3.1 侧边栏

- 增加“创作”分组及三个导航项。
- “游乐场”继续指向现有 `/console/playground`。
- “图像”指向 `/console/image`。
- “视频”指向 `/console/video`。
- 沿用现有 sidebar 模块可见性机制；未配置新权限时，图像和视频默认随创作分组显示。
- 当前选中项、折叠状态、移动端抽屉行为与现有导航一致。

### 3.2 图像页面

移植 `gpt_image_playground` 的核心组件和状态：提示词输入、参数面板、参考图/遮罩、生成任务卡片、历史记录、收藏、详情和下载。项目本身使用 IndexedDB 保存历史，第一阶段沿用该行为。

API 配置提供当前 New API 同源地址作为默认值，允许用户临时修改地址和密钥。密钥仅在浏览器端保存并用于请求，后续第二阶段将替换为服务端自动密钥提供器。

### 3.3 视频页面

移植 `alasano/sora-2-playground` 的创建表单、模型/尺寸/时长选择、参考素材、任务进度、视频输出、历史、下载和 Remix。去除 Next.js 页面壳和服务端 API 路由，保留 React 组件与 IndexedDB 历史。当前 New API 没有 OpenAI 兼容的视频删除路由，因此第一阶段的“删除历史”只删除浏览器本地记录，不承诺删除上游任务。

视频服务层使用 New API 已存在的接口：

- `POST /v1/videos`：创建任务。
- `GET /v1/videos/:id`：查询任务状态。
- `GET /v1/videos/:id/content`：读取视频内容。
- `POST /v1/videos/:id/remix`：创建 Remix 任务。

服务层统一处理 OpenAI 兼容响应与 New API 错误格式，页面不直接依赖 OpenAI 官方域名。

## 4. 数据流与认证

1. 用户先通过现有 New API 登录并进入受保护的 console 路由。
2. 首次进入图像或视频页时，读取该 Playground 的本地配置；没有配置则显示其设置对话框，不发起生成请求。
3. 用户提交任务时，浏览器携带当前配置的 token 调用同源 `/v1` 接口。
4. New API 按现有 `TokenAuth`、模型分发、限流和计费链路处理请求。
5. 图像结果与视频历史保存在浏览器 IndexedDB；生成中的视频任务 ID 持久化，以便刷新后继续轮询。
6. 第二阶段只替换第 2-3 步的凭证来源，页面组件和 API 服务接口保持不变。

第一阶段明确接受“密钥在浏览器配置中可见”的临时安全模型，仅建议在受控账号和 HTTPS 下使用；不得把任何管理员密钥写入前端代码、默认配置、URL 或镜像环境变量。

## 5. 错误处理

- 未配置密钥：显示设置对话框和当前接口地址，不显示空白页面。
- 401/403：提示凭证无效或无权限，并提供重新配置入口。
- 404/模型不支持：显示具体模型和接口提示，保留用户输入。
- 429/配额不足：显示限流或额度错误，避免自动快速重试。
- 视频轮询失败：按有限次数退避重试，超过上限将任务标记失败并保留错误信息。
- 下载失败：不清除历史记录，允许用户重试或复制任务信息。
- 删除视频历史：只删除浏览器 IndexedDB 记录，并在确认对话框中明确不会删除上游任务。
- 组件异常：使用现有错误边界隔离创作页面，不能影响其他 console 路由。

## 6. 验证与验收

实现后必须完成：

- 图像页和视频页的前端构建通过。
- 现有聊天 Playground、登录、控制台和侧边栏原有入口回归检查通过。
- 桌面与窄屏移动端检查：导航不重叠，输入区、结果区和滚动区域可用。
- 使用一个具备相应模型权限的普通 New API token 完成图像任务提交和视频任务创建/轮询/播放或明确验证到上游模型限制。
- 未配置、无效 token、401/429 和上游错误均显示可理解的页面反馈。
- 生产部署只保留 New API 与 PostgreSQL 两个长期服务；构建采用串行/限内存方式，部署后检查健康接口和三个创作路由。
- 保留当前生产镜像标签，若新版本启动失败可回滚到上一镜像。

## 7. 部署与后续扩展

前端构建产物合入 New API 镜像，服务器不运行额外 Node/Python Playground 进程。部署前先构建并运行本地 smoke test，再替换服务器镜像；PostgreSQL 数据卷和现有配置不变。

第二阶段增加按用户/工具维度的专用 token 创建与复用服务，浏览器只获得短期会话授权或通过同源后端代理访问，不把长期 token 放在 URL。该阶段应单独设计权限、回收、并发创建幂等和审计日志。
